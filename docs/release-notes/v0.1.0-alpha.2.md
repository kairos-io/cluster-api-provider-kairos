# v0.1.0-alpha.2 — Release Notes

This is a pre-release. The API surface, defaults, and on-disk layout may
change before v0.1.0 ships.

Read the [upgrade guide](../UPGRADING.md) before installing. If you are
upgrading from v0.1.0-alpha.1, the breaking-changes section below is required
reading.

---

## Breaking Changes

### 1. No-credential KairosConfig is now rejected

The validating webhook now rejects any `KairosConfig` or `KairosConfigTemplate`
that carries no credential (commit 5beb8ae). A credential means at least one of:
`userPassword`, `userPasswordSecretRef`, `sshPublicKey`, or `githubUser`.

The previous implicit behavior (no credential → the controller silently injected
`kairos:kairos` as user/password) is gone. Alpha-1 manifests that omitted all
credential fields will fail webhook admission after this upgrade.

**Required operator action before upgrade:** add a `userPasswordSecretRef`
(recommended), `sshPublicKey`, or `githubUser` to every `KairosConfig` and
`KairosConfigTemplate`. See the [credential migration row in UPGRADING.md](
../UPGRADING.md#credential-migration-default-credentials-removed).

### 2. KCP no longer auto-discovers the control-plane endpoint (KD-12)

`KairosControlPlane` no longer writes `Cluster.Spec.ControlPlaneEndpoint` from
Machine IP addresses (CAPV/CAPD) or from the LoadBalancer Service ingress
(CAPK). Writing that field is the infra provider's responsibility per the CAPI
v1beta2 contract (commit 95b8630).

**Effect:** if your CAPV, CAPD, or CAPK cluster's
`<Infra>Cluster.Spec.ControlPlaneEndpoint` is unset, `KairosControlPlane`
stalls with `Available=False(WaitingForInfrastructureControlPlaneEndpoint)`
after this upgrade.

**Required operator action:** ensure `<Infra>Cluster.Spec.ControlPlaneEndpoint`
is set before or at apply time. All sample manifests already include a
`controlPlaneEndpoint` stanza. See the migration table in
[UPGRADING.md](../UPGRADING.md#kd-12-kcp-no-longer-writes-clusterspeccontrolplaneendpoint).

### 3. Mid-bootstrap alpha-1 clusters require a rollout after upgrade

Any control-plane node whose bootstrap Secret was rendered by the alpha-1
controller (no node-push block) must receive a new bootstrap Secret before the
controller can acquire the workload kubeconfig. Trigger a KCP rollout (or
delete the KairosConfig) to re-render the Secret with the alpha-2 template.
Clusters that are already healthy and whose kubeconfig Secret is present are
unaffected. See [UPGRADING.md §KD-3b](../UPGRADING.md#kd-3b-ssh-eliminated-from-controller-normal-operation).

## New Features

### Metal3 / CAPM3 bare-metal support (Phase 4)

Single-node k0s and k3s clusters on bare-metal nodes managed by Cluster API
Provider Metal3. Lab-validated end-to-end (see e2e matrix below). Includes:

- Post-bootstrap k0s providerID patch via config-drive metadata, set after
  k0s starts (KD-51).
- Fail-open non-Metal3 path (KD-55): if providerID cannot be determined at
  render time, the node joins without it and the in-VM script retries.
- Hadron (musl-libc Kairos) validated on CAPM3 with both k0s and k3s.

See `docs/QUICKSTART_CAPM3.md` and `config/samples/capm3/`.

The prior "alpha-2 HOLD on Hadron open-vm-tools" is lifted. All four matrix
cells (CAPV+k0s, CAPV+k3s, CAPM3+k0s, CAPM3+k3s) are GREEN on Hadron. CAPK
is excluded from the alpha-2 regression matrix; it passed alpha-1 and has no
code changes targeted at it in this release.

**Installed-disk requirement:** Ironic must deploy a fully-installed Kairos
raw disk image (not an auroraboot auto-reset image). See
[QUICKSTART_CAPM3.md §Building the disk image](QUICKSTART_CAPM3.md#building-the-disk-image)
for the build pipeline and Hadron-specific failure modes.

### Opt-in host-key-verified SSH fallback for air-gapped environments (SSHFallback)

`KairosControlPlane.spec.sshFallback` enables a controller-side SSH path that
fires after a configurable timeout when the node-push path is unreachable
(air-gapped CAPV or CAPM3). Host-key verification is mandatory — there is no
trust-on-first-use mode.

Configuration: create a `kubernetes.io/ssh-auth` Secret (private key) and an
Opaque Secret (`known_hosts`), then set:

```yaml
spec:
  sshFallback:
    enabled: true
    identitySecretRef:
      name: kairos-ssh-identity
    knownHostsSecretRef:
      name: kairos-ssh-known-hosts
    activateAfter: 15m   # must exceed the 10m Info→Warning threshold
```

See [QUICKSTART_CAPV.md §Air-gapped fallback](QUICKSTART_CAPV.md#air-gapped-fallback-sshfallback)
and [QUICKSTART_CAPM3.md §Air-gapped fallback](QUICKSTART_CAPM3.md#air-gapped-fallback-sshfallback)
for worked examples.

### KairosControlPlaneTemplate validating/mutating webhook

The `KairosControlPlaneTemplate` resource now has its own validating and
mutating webhook. Previously only `KairosControlPlane` was webhook-validated;
templates could be created with fields that would be rejected at expansion time.

### `spec.files` rendered into cloud-config

`KairosConfig.spec.files` (and the equivalent in `KairosConfigTemplate`) was
accepted and stored but silently ignored by the bootstrap renderer in alpha-1.
Files are now written to the node's filesystem via the cloud-config
`write_files:` list at bootstrap time, on all distributions (k0s, k3s) and all
infrastructure providers (CAPV, CAPK, CAPD, CAPM3).

Validation rejects non-absolute paths, `..` traversal segments, non-octal
permissions, and malformed `user:group` owners. See [API_REFERENCE.md §Writing
files to nodes](API_REFERENCE.md#writing-files-to-nodes) for limits and the
static-IP caveat when using `spec.files` for systemd-networkd configuration on
pre-installed images.

### Persistent-state paths via cloud-config

The provider injects `/system/oem/12_kairos-capi-persistency.yaml` into every
node's cloud-config to declare persistent paths across reboots and A/B upgrades
(KD-23). Paths include `/etc/k0s`, `/var/lib/k0s`, `/etc/rancher`,
`/var/lib/rancher`, `/etc/cni`, `/etc/systemd`, `/etc/ssh`, and others. The
mechanism uses immucore's `extra-layout.env` and is additive — it does not
overwrite the stock image's persistent paths.

## Security Hardening

- **Default credentials removed.** The implicit `kairos:kairos` user/password
  and `PasswordAuthentication yes` SSH config are gone. `KairosConfig` with no
  credential is now webhook-rejected. All sample manifests use
  `userPasswordSecretRef`.

- **Hostname RCE closed; shell-context fields shquote'd.** Hostname and other
  fields that land in shell contexts are now shell-quoted in the cloud-config
  renderer to prevent injection via crafted field values.

- **SSH removed from controller hot path.** The controller no longer SSHes into
  nodes to retrieve the workload kubeconfig or patch `Node.Spec.ProviderID`.
  Both operations are now owned by the node (node-push for kubeconfig; in-VM
  systemd service for providerID). The controller's attack surface is reduced
  to management-cluster API calls only during normal operation.

- **RBAC minimization.** Dropped the `infrastructure.cluster.x-k8s.io` wildcard
  resource. Enumerated kinds including Metal3 resources. Per-cluster node SA
  for the node-push kubeconfig path scopes token authority to a single cluster.

## CAPI v1beta2 Contract

- KCP no longer writes `Cluster.Spec.ControlPlaneEndpoint` (KD-12, commit
  95b8630) — the infra provider owns it. KCP stalls with
  `Available=False(WaitingForInfrastructureControlPlaneEndpoint)` until the
  field is populated.

- v1beta2 status fields (`initialization.controlPlaneInitialized`,
  `initialization.dataSecretCreated`) are present.

- `status.v1beta2.conditions` is absent (KD-13, known limitation).

## Bug Fixes

- **`spec.files` rendered into cloud-config.** See New Features above.

- **k0s providerID via post-bootstrap patch (KD-51 redesign).** On Metal3 the
  BareMetalHost UUID is not known at bootstrap render time. The provider now
  patches providerID after bootstrap via an in-VM systemd service. The
  non-Metal3 path is fail-open (KD-55).

- **k0s Metal3 `--kubelet-extra-args` quoting.** Shell splitting on providerID
  values that contain special characters is fixed with explicit quoting in the
  cloud-config stage.

- **Latched-failure recovery (KD-14).** Bootstrap and KCP controllers now clear
  `status.failureReason` and `status.failureMessage` on every successful
  reconcile. Previously a transient failure (for example, a missing Secret)
  could latch these fields in a terminal-looking state and block the CAPI
  Machine controller from proceeding even after the condition was resolved.

- **`patch.Helper` created before early returns (KD-14).** `observedGeneration`
  and conditions are now flushed on paused, no-owner, and no-Cluster early-exit
  paths in both controllers.

## Known Limitations

- **HA control planes** (`replicas > 1`). The validating webhook rejects values
  greater than 1. Tracked as KD-5 / KD-25.

- **`spec.preCommands` / `spec.postCommands`** are accepted by the API but not
  yet rendered into the cloud-config. See API reference.

- **`status.v1beta2.conditions`** is absent from both `KairosConfig` and
  `KairosControlPlane` status. The `conditions` array (v1beta1-style) is
  present and populated. Tracked as KD-13.

- **Static-IP race on pre-installed images.** Writing a systemd-networkd
  `.network` file via `spec.files` does not take effect on the running boot
  if networkd already holds a DHCP lease. Use infrastructure-layer IP
  management (DHCP reservation or IPAM) instead. See
  [API_REFERENCE.md §Static IP via systemd-networkd](API_REFERENCE.md#static-ip-via-systemd-networkd--known-limitation-on-pre-installed-images).

## What Was Fixed Relative to the alpha-1 Known Limitations

The alpha-1 release notes carried: "`reconcileDelete` does NOT clean up child
resources on `Cluster` deletion. Manual cleanup required."

This limitation is resolved in alpha-2 (commit 5112019, KD-4). The bootstrap
controller now deletes the owned bootstrap Secret before removing its finalizer.
The KCP controller now drains owned Machines (with UID-scoped ownership checks
to avoid false matches) before removing its finalizer. Manual cleanup of child
resources is no longer required for normal cluster deletion.
