# v0.1.0-alpha.2 — release notes

This is a pre-release. The API surface, defaults, and on-disk layout may
change before v0.1.0 ships.

---

## Breaking changes

None relative to v0.1.0-alpha.1.

---

## New features

- **Metal3 / CAPM3 support (Phase 4).** Single-node k0s and k3s clusters
  on bare-metal nodes managed by Cluster API Provider Metal3. Includes the
  providerID post-bootstrap patch for k0s (KD-51) and the fail-open
  non-Metal3 path (KD-55). See `docs/QUICKSTART_CAPM3.md` and
  `config/samples/capm3/`.

---

## Bug fixes

- **`spec.files` is now rendered into the cloud-config.** `KairosConfig.spec.files`
  (and `KairosConfigTemplate.spec.template.spec.files`) was accepted by the
  API server and stored in etcd, but was silently ignored by the bootstrap
  renderer. Files are now written to the node's filesystem via the cloud-config
  `write_files:` list at bootstrap time, on all distributions (k0s, k3s) and
  all infrastructure providers (CAPV, CAPK, CAPD, CAPM3). Validation rejects
  non-absolute paths, `..` traversal segments, non-octal permissions, and
  malformed `user:group` owners. See `docs/API_REFERENCE.md` §
  "Writing files to nodes" for field limits and the static-IP caveat when
  using `spec.files` for systemd-networkd configuration on pre-installed
  images.

- **k0s providerID via post-bootstrap patch (KD-51 redesign).** Previously
  the k0s providerID was injected into the kubelet args at render time; on
  Metal3 this is unavailable at bootstrap because the BareMetalHost UUID is
  not known until provisioning completes. The provider now patches the providerID
  after bootstrap via a post-boot mechanism. The non-Metal3 path is fail-open
  (KD-55): if the providerID cannot be determined at bootstrap time, the node
  joins without it and relies on the infrastructure provider to supply it.

- **k0s Metal3 `--kubelet-extra-args` quoting.** The `--kubelet-extra-args`
  flag was not quoted when the providerID contained characters that the shell
  would split. Fixed with explicit quoting in the cloud-config stage.

---

## Security

No security-relevant changes in this release beyond the providerID/kubelet-args
quoting fix noted above.

---

## Internal

- Cloud-config static-IP caveat documented for `spec.files` and systemd-networkd
  on pre-installed images (no code change; user-facing guidance only).

---

## What does not work yet

- **HA control planes** (`replicas > 1`). The validating webhook rejects values
  greater than 1. Tracked as KD-5 / KD-25.
- **`reconcileDelete`** does not clean up child resources on `Cluster` deletion.
  Manual cleanup required.
- **`spec.preCommands` / `spec.postCommands`** are accepted by the API but not
  yet rendered into the cloud-config. See API reference.
